### 《参考解析》

1. **Redis 大 Key 的危害与解决**：大 Key（如超大 Hash、List、单个 Value 过大的 String）会导致单次操作耗时长阻塞主线程（Redis 单线程模型下影响尤其明显）、内存分布不均引发数据倾斜、以及主从同步/迁移时网络阻塞。解决思路是拆分大 Key（如按业务维度拆成多个小 Key）、用 `SCAN` 替代 `KEYS` 做批量处理避免阻塞、对已存在的大 Key 定期做异步拆分迁移。
2. **AOP 的实现原理**：Spring AOP 基于动态代理实现——目标类实现了接口时使用 JDK 动态代理，没有实现接口时使用 CGLIB 生成子类代理；在代理对象方法执行前后织入切面逻辑（如打印调用链日志、记录耗时），从而做到业务代码无侵入。要打通完整调用链，通常会在切面里生成/透传一个 TraceId，并借助 ThreadLocal 或 MDC 把 TraceId 写入日志上下文，实现跨方法甚至跨服务的链路追踪。
3. **线程阻塞排查思路**：可以用 `jstack` 导出线程栈，查找处于 `BLOCKED` 或长时间 `WAITING` 状态的线程，结合线程名和堆栈定位具体阻塞点；也可以借助 Arthas 的 `thread` 命令实时查看 CPU 占用最高的线程和其调用栈，快速定位是锁竞争、IO 等待还是死循环导致的阻塞。
