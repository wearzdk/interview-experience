### 《参考解析》

1. **动态线程池热更新**：核心思路是把线程池参数（核心数、最大数、队列容量、拒绝策略）托管到配置中心（如 Nacos），线程池实例持有一个可变的委托对象，监听配置变更事件后通过 `setCorePoolSize()` / `setMaximumPoolSize()` 等方法动态调整，无需重启应用即可生效；调整前要注意队列已有任务和当前活跃线程数的边界情况，避免瞬时收缩导致任务被拒绝。
2. **MySQL 索引下推与覆盖索引**：索引下推（ICP）是指在联合索引遍历过程中，将 WHERE 条件中能通过索引字段判断的部分提前在存储引擎层过滤，减少回表次数；覆盖索引则是查询所需的所有列都能在索引中直接取到，无需回表查主键索引。最左匹配原则失效常见于对索引列使用函数、隐式类型转换、`OR` 连接非索引列，或范围查询之后的列。
3. **数据库与缓存一致性**：业界主流是 Cache Aside（旁路缓存）模式，更新时"先更新数据库、再删除缓存"；为应对删除失败或并发脏读，可搭配延时双删或监听 binlog（如 Canal）异步淘汰缓存做兜底，强一致性场景则需要引入分布式事务或读写穿透（Write/Read Through）方案。
4. **延时消息实现**：常见方案一是使用消息队列自带的延时/定时消息能力（如 RocketMQ 的定时消息、RabbitMQ 死信队列 + TTL）；方案二是自行维护一个基于时间戳排序的 ZSet，用异步线程定时轮询到期任务再触发投递，适合延时时间不固定的场景。
